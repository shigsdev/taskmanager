Task Manager — Architecture

Your task manager is a single-user web app that lives at web-production-3e3ae.up.railway.app. It stores your tasks, goals, and projects in a database, emails you a daily digest, and runs a few automated jobs in the background.

This page explains how it's built in plain English. Each section also has an "Engineering" collapsible for the technical detail (also useful for any engineer reading the repo). For how to USE the app, see Help.

The same six pieces, written out:

  1. The web app — a Python service (Flask) that serves you the HTML pages and handles your clicks. Runs inside Gunicorn workers on Railway.
  2. The database — Postgres (managed by Railway). Stores your tasks, goals, projects, etc. See the schema below.
  3. Scheduled jobs — four automated tasks that run on a clock (in your timezone, default America/New_York):
    • 06:30 daily — sends you the digest email
    • 00:01 nightly — moves Tomorrow tasks to Today
    • 00:05 nightly — spawns the day's recurring tasks
    • every 45 seconds — heartbeat (proves the scheduler is alive to other workers)
  4. Authentication — Google sign-in. Only one email is allowed in (the one configured on Railway); everyone else gets a 403 page. There's also a long-lived signed "validator cookie" used by automated post-deploy checks.
  5. External services — OpenAI Whisper (transcribes voice memos), Anthropic Claude (extracts tasks from voice memos and image scans), Google Vision (OCRs images), SendGrid (delivers the digest email).
  6. Hosting — Railway. HTTPS enforced. Health checks every few seconds via /healthz.

Step by step:

  1. At 00:05 every night, an automated job wakes up.
  2. It calculates today's date in your timezone (default America/New_York — using UTC would put 10pm-ET spawns on tomorrow's date).
  3. It looks up which recurring task templates are scheduled to fire today (e.g. a weekly "Walk dog" template is included on Tuesdays only).
  4. For each template, it checks whether you already have a task for today's date — across all tiers, not just Today. So if you'd already moved "Walk dog" into "this week" with today's due date, the job sees it and skips spawning a duplicate.
  5. For the rest, it creates a new task in the Today tier with today's due date.
  6. If the recurring template has subtasks, it creates those too (also in Today).
  7. You wake up, open the app, and the day's recurring tasks are waiting in Today.

Step by step:

  1. You hit the 🎙️ voice memo icon on the capture bar and start talking.
  2. Your browser records the audio (10-minute hard cap).
  3. When you stop, the audio file uploads to the server.
  4. The server checks the file (correct format, not too big, not empty), then forwards it to OpenAI's Whisper for transcription.
  5. Whisper sends back the transcript as plain text.
  6. The server sends the transcript to Anthropic's Claude, along with a list of your active project and goal titles, asking it to identify discrete tasks and infer metadata.
  7. Claude returns a list of candidates. The server resolves any project/goal hints to real IDs (case-insensitive title match).
  8. You see a review screen: each candidate is an editable row. Reflections (non-tasks) are tucked into a collapsed section at the bottom, unchecked by default.
  9. You uncheck what you don't want, edit anything wrong, then hit Confirm.
  10. The server creates real tasks in your database with the fields you confirmed.

The same logic, written out:

1. Is this local development with the bypass turned on?

YES → skip authentication entirely. Log you in as the configured email. Used only on your local dev machine when the .env.dev-bypass file exists, AND no RAILWAY_* tripwire env var is set. Four safety gates total.

NO → continue to check #2.

2. Do you have a "validator cookie" AND is this a safe HTTP method?

A "validator cookie" is a long-lived signed token used by the post-deploy automation to read pages without OAuth. "Safe" methods = GET, HEAD, OPTIONS.

YES, AND the email in the cookie matches your authorized email → log you in via the cookie. Used by the post-deploy Playwright tests.

NO (cookie missing/invalid OR you're trying to mutate data) → continue to check #3.

3. Do you have a Google OAuth session?

NO session → redirect you to Google, then back to the page you wanted.

YES session, AND the Google account email matches your authorized email → log you in. (Most normal use is this path.)

YES session, but the email doesn't match → 403 Unauthorized page (the single-user lockdown).

{# ---- Post-deploy verification ---- #}

Post-deploy verification

After git push Railway builds + rolling- deploys. Two checks run against the live URL to confirm the new container is actually serving correctly.

  1. python3.14 scripts/validate_deploy.py polls /healthz every 15s for up to 10 minutes. Deploy is GREEN only when all of: SHA match (proves you're hitting the NEW container, not the old one), HTTP 200, all 9 health checks ok, auth preflight via the long-lived validator cookie, AND a clean error-log scan since the new container's started_at (the since-filter param- name bug from #49 is fixed).
  2. npm run test:e2e:prod runs 5 Playwright smoke tests against the live Railway URL using the validator cookie: auth status, home page renders w/o JS errors, goals page same, /api/tasks shape, /healthz reports git_sha.
  3. Only when both are GREEN does the BACKLOG row flip from 🔄 IN PROGRESS to ✅ — and that flip is a separate commit so the SHA history reads "ship, verify, mark done" in three discrete steps.

Known testing limitations

Things the local + post-deploy gates cannot catch. Surfacing them honestly so future investigation knows where to look when a bug "should" have been caught.

LimitationWhyMitigation
SQLite vs Postgres Local tests use SQLite (no native enum type, no concurrent locking, no pg-specific catalogs). Production is Postgres — strict enums, real concurrency. The enum_coverage healthz check (#53) catches the most common class. Post-deploy automation surfaces other Postgres-only issues at deploy time.
Validator cookie is GET-only The post-deploy automation's auth token is intentionally narrow — it can't POST/PATCH/DELETE so a leaked cookie can't mutate data (ADR-004). Post-deploy smoke covers READ paths only. Mutating endpoints are exercised by local Playwright (which uses the dev bypass) + manual Phase 6 regression.
Local Playwright fails on Mac Node TLS handshake mis-negotiates with Playwright's Azure CDN from this Mac specifically (#41). Doesn't affect prod or Windows. Mac contributors run gate 4 + prod e2e from the Windows desktop instead. Documented; not blocking.
External API success is mocked Tests don't actually call SendGrid / Whisper / Claude / Vision — would burn budget and be flaky. Mocked at the egress.safe_call_api seam. The Settings → SendGrid Test send button (#48) does a real round-trip when needed. Other services exercised when actually used.
HTML routes still use Flask's default error page The global error handler (#50) returns JSON only for /api/ paths to avoid breaking the browser experience on HTML routes. Acceptable — most failures the user sees are API calls. HTML route failures are rarer; address with a custom error template if/when needed.
Single-user app — multi-tenant assumptions untested The AUTHORIZED_EMAIL single-user lockdown (ADR-001) assumes one user; rate limiting is per-process; recovery is manual restore from Railway backups. Out of threat model — see CLAUDE.md threat-model section. Revisit if multi-user is ever added.
Engineering: how to actually run the gates + cascade rules

Run all 11 gates locally:

bash scripts/run_all_gates.sh

Single command, exits 0 only if all pass. The script auto-manages the bypass server for gate 4 and tears it down on exit. Skipping a gate is an SOP violation; if a gate genuinely cannot run in this environment (e.g. node missing on a Mac without node dev setup), the commit message must include a Gates-skipped: <gate> (<reason>) trailer.

Post-deploy validation:

python3.14 scripts/validate_deploy.py
export TASKMANAGER_SESSION_COOKIE="$(cat ~/.taskmanager-session-cookie)"
npm run test:e2e:prod

CLAUDE.md cascade-check rules (auto-applied on every change — see CLAUDE.md):

  • New env var → README env-var table; docs_sync_check enforces
  • New route / scheduler job / API endpoint → ARCHITECTURE.md; arch_sync_check enforces
  • New file-upload endpoint → use utils.validate_upload (ADR-025); add oversize/empty/MIME tests
  • New external API caller → route through egress.safe_call_api (ADR-023); add scrub-key test
  • New db.Model subclass → add to _ER_TABLE_GROUPS + _ER_TABLE_ORDER + _SCHEMA_DESCRIPTIONS; drift-gate tests enforce (#43, #44, #53)
  • New column on existing model → add description to _SCHEMA_DESCRIPTIONS (#44); drift-gate enforces
  • New gate added to run_all_gates.sh → also update this section of /architecture (#54)
  • User-visible behavior change → update templates/docs.html (Help) per the user-facing fact-check SOP
