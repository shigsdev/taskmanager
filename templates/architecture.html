{# ============================================================= PROCESS FLOW — recurring spawn (#34, #35, #38) ============================================================= #}

Process flow: recurring task spawn

The recurring_spawn APScheduler cron fires at 00:05 in DIGEST_TZ (default America/New_York) and materialises any RecurringTask firing today as a real Task in the Today tier. Idempotent across all active tiers — see ADR-026 for the (recurring_task_id, due_date) dedup key that closes the cross-tier duplicate gap (#38). 

sequenceDiagram
    autonumber
    participant Cron as APScheduler
(recurring_spawn @ 00:05 DIGEST_TZ)
    participant Service as recurring_service
spawn_today_tasks()
    participant DB as Postgres
(tasks, recurring_tasks)
    Cron->>Service: spawn_today_tasks()
    Service->>Service: target_date = _local_today_date() (#28)
    Service->>DB: SELECT active RecurringTask WHERE _template_fires_on(today)
    DB-->>Service: templates due today
    Service->>DB: SELECT active Task WHERE recurring_task_id IN (...) AND due_date = today
    DB-->>Service: existing_keys (cross-tier dedup, #38)
    loop for each template
        alt (rt.id, target_date) in existing_keys
            Service->>Service: skip (already spawned or planned-ahead)
        else
            Service->>DB: INSERT Task(tier=TODAY, due_date=today, recurring_task_id=rt.id)
        end
    end
    Service->>DB: COMMIT (parents)
    Service->>DB: INSERT subtasks from rt.subtasks_snapshot (#26)
    Service->>DB: COMMIT (subtasks)
    Service-->>Cron: spawned tasks

Related ADRs: ADR-013 (subtasks_snapshot JSON column), ADR-014 (Tomorrow tier + auto-roll cron), ADR-016 (tier→due_date auto-fill), ADR-026 (spawn due_date + cross-tier dedup).

{# ============================================================= PROCESS FLOW — voice memo capture (#36, #37) ============================================================= #}

Process flow: voice memo capture

Browser records audio via MediaRecorder, POSTs to the server, server forwards to OpenAI Whisper for transcription, then to Anthropic Claude with a structured prompt that returns task candidates with title / type / tier / due_date / project_hint / goal_hint / is_task. Hints are resolved against the user's actual projects + goals via case-insensitive exact-title match. User reviews on screen, edits if needed, confirms to create real Tasks. 

sequenceDiagram
    autonumber
    participant Browser as Browser
(MediaRecorder + voice_memo.js)
    participant Voice as voice_api
POST /api/voice-memo/parse
    participant Whisper as OpenAI Whisper
(via egress.safe_call_api)
    participant Scan as scan_service
parse_voice_memo_to_tasks()
    participant Claude as Anthropic Claude
(via egress.safe_call_api)
    participant DB as Postgres
    Browser->>Browser: record audio (10-min cap)
    Browser->>Voice: POST audio (multipart, audio/webm or audio/mp4)
    Voice->>Voice: utils.validate_upload (MIME, size, empty)
    Voice->>Whisper: POST /audio/transcriptions
    Whisper-->>Voice: transcript
    Voice->>Scan: parse_voice_memo_to_tasks(transcript)
    Scan->>DB: SELECT active Project + Goal titles (for hint resolution)
    Scan->>Claude: POST /v1/messages with structured prompt + titles
    Claude-->>Scan: candidates JSON
    Scan->>Scan: _normalise_voice_candidates (resolve hints to UUIDs)
    Scan-->>Voice: candidates
    Voice-->>Browser: 200 OK with candidates
    Browser->>Browser: render review screen (tier select, date input,
project + goal dropdowns, is_task=false → Reflections section)
    Browser->>Voice: POST /api/voice-memo/confirm with selected candidates
    Voice->>Scan: create_tasks_from_candidates()
    Scan->>DB: INSERT Task(s) with project_id, goal_id, tier, due_date
    Voice-->>Browser: 201 Created

Related ADRs: ADR-005 (voice memo error handling + iOS Safari Content-Type quirk), ADR-020 (NLP MVP — structured prompt grammar), ADR-021 (NLP phase 2 — hint resolution + non-task detection), ADR-023 (egress module), ADR-025 (upload validation helper).

{# ============================================================= PROCESS FLOW — auth (ADRs 001, 002, 003, 004) ============================================================= #}

Process flow: authentication

Three authentication paths, in priority order: dev bypass (local only, four-gate check), validator cookie (long-lived signed token, GET-only routes), then Google OAuth (the production user path). All paths end with the same AUTHORIZED_EMAIL single- user lockdown check. 

sequenceDiagram
    autonumber
    participant Browser
    participant Flask as Flask app
@login_required wrapper
    participant Bypass as _dev_bypass_active()
(local only, four-gate)
    participant Validator as validator_cookie
(signed long-lived token)
    participant OAuth as Google OAuth
(Flask-Dance)
    participant View as protected view
    Browser->>Flask: GET /protected
    Flask->>Bypass: check 4-gate dev bypass
    alt dev bypass active (LOCAL ONLY)
        Bypass-->>Flask: short-circuit, AUTHORIZED_EMAIL
        Flask->>View: call view(email=AUTHORIZED_EMAIL)
    else validator_token cookie + safe HTTP method (GET/HEAD/OPTIONS)
        Flask->>Validator: verify signed token
        alt valid + email matches AUTHORIZED_EMAIL
            Validator-->>Flask: validator_email
            Flask->>View: call view(email=validator_email)
        else invalid OR mutation method
            Flask->>OAuth: continue to OAuth path
        end
    else
        Flask->>OAuth: get_current_user_email()
        alt no session
            OAuth-->>Browser: 302 redirect to /login/google
            Browser->>OAuth: complete OAuth flow
            OAuth-->>Browser: redirect back to /protected
        else session exists
            OAuth-->>Flask: email from token
            alt email != AUTHORIZED_EMAIL
                Flask-->>Browser: 403 unauthorized.html
            else
                Flask->>View: call view(email=email)
            end
        end
    end
    View-->>Browser: response

Related ADRs: ADR-001 (single-user OAuth lockdown), ADR-002 (four-gate dev bypass), ADR-003 (validator cookie narrow scope, SUPERSEDED), ADR-004 (broaden validator cookie to GET-only routes).
