{# ============================================================= PROCESS FLOW — recurring spawn (#34, #35, #38) ============================================================= #}

Process flow: recurring task spawn

The recurring_spawn APScheduler cron fires at 00:05 in DIGEST_TZ (default America/New_York) and materialises any RecurringTask firing today as a real Task in the Today tier. Idempotent across all active tiers — see ADR-026 for the (recurring_task_id, due_date) dedup key that closes the cross-tier duplicate gap (#38).

Linear flow with no branches — rendered as a numbered list (#43). For flows with conditionals, see voice memo + auth below which use Mermaid sequence diagrams.

  1. APScheduler fires the recurring_spawn job at 00:05 local (DIGEST_TZ) and calls spawn_today_tasks() with no arguments.
  2. The service computes target_date in the user's timezone via _local_today_date() (ADR-016) — using UTC would put 10pm-ET spawns on tomorrow's date.
  3. SELECT active RecurringTask rows where the template fires today (frequency check via _template_fires_on()).
  4. SELECT active Task rows already linked to those templates with due_date == today, across all tiers. This is the cross-tier dedup that closes Gap B from ADR-026 (a planned-ahead "Meds" task in this_week for Friday blocks the Friday spawn).
  5. For each template not in the existing-keys set, INSERT a new Task with tier=TODAY, due_date=target_date, and recurring_task_id=template.id. COMMIT.
  6. For each spawned parent that has a subtasks_snapshot on its template (ADR-013), INSERT one subtask row per snapshot entry, also with due_date=target_date. COMMIT.
  7. Return the list of spawned tasks (parents only) to the cron — used for the response body of the manual /api/recurring/spawn endpoint.

Related ADRs: ADR-013 (subtasks_snapshot JSON column), ADR-014 (Tomorrow tier + auto-roll cron), ADR-016 (tier→due_date auto-fill), ADR-026 (spawn due_date + cross-tier dedup).

{# ============================================================= PROCESS FLOW — voice memo capture (#36, #37) ============================================================= #}

Process flow: voice memo capture

Browser records audio via MediaRecorder, POSTs to the server, server forwards to OpenAI Whisper for transcription, then to Anthropic Claude with a structured prompt that returns task candidates with title / type / tier / due_date / project_hint / goal_hint / is_task. Hints are resolved against the user's actual projects + goals via case-insensitive exact-title match. User reviews on screen, edits if needed, confirms to create real Tasks. 

sequenceDiagram
    autonumber
    participant Browser
    participant Voice as voice_api
    participant Whisper
    participant Scan as scan_service
    participant Claude
    participant DB
    Browser->>Browser: record audio (10-min cap)
    Browser->>Voice: POST audio (multipart)
    Voice->>Voice: validate_upload (MIME, size, empty)
    Voice->>Whisper: POST /audio/transcriptions
    Whisper-->>Voice: transcript
    Voice->>Scan: parse_voice_memo_to_tasks(transcript)
    Scan->>DB: SELECT active Project + Goal titles
    Scan->>Claude: POST /v1/messages with prompt + titles
    Claude-->>Scan: candidates JSON
    Scan->>Scan: resolve hints to UUIDs
    Scan-->>Voice: candidates
    Voice-->>Browser: 200 OK with candidates
    Browser->>Browser: render review screen
    Browser->>Voice: POST /api/voice-memo/confirm
    Voice->>Scan: create_tasks_from_candidates()
    Scan->>DB: INSERT Task(s)
    Voice-->>Browser: 201 Created

Participant labels in the diagram are short for layout. Real callers (#43): Browser = MediaRecorder + static/voice_memo.js. Voice = voice_api.py, route POST /api/voice-memo/parse. Whisper = OpenAI Whisper at api.openai.com/v1/audio/transcriptions, called via egress.safe_call_api. Scan = scan_service.py, function parse_voice_memo_to_tasks. Claude = Anthropic Claude at api.anthropic.com/v1/messages, also via egress.safe_call_api. DB = Postgres. The review screen renders a tier select, date input, project + goal dropdowns; is_task=false items go to a collapsed "Reflections" section.

Related ADRs: ADR-005 (voice memo error handling + iOS Safari Content-Type quirk), ADR-020 (NLP MVP — structured prompt grammar), ADR-021 (NLP phase 2 — hint resolution + non-task detection), ADR-023 (egress module), ADR-025 (upload validation helper).

{# ============================================================= PROCESS FLOW — auth (ADRs 001, 002, 003, 004) ============================================================= #}

Process flow: authentication

Three authentication paths, in priority order: dev bypass (local only, four-gate check), validator cookie (long-lived signed token, GET-only routes), then Google OAuth (the production user path). All paths end with the same AUTHORIZED_EMAIL single- user lockdown check. 

sequenceDiagram
    autonumber
    participant Browser
    participant Flask
    participant Bypass
    participant Validator
    participant OAuth
    participant View
    Browser->>Flask: GET /protected
    Flask->>Bypass: check dev bypass
    alt dev bypass active (LOCAL ONLY)
        Bypass-->>Flask: AUTHORIZED_EMAIL
        Flask->>View: view(email=AUTHORIZED_EMAIL)
    else validator cookie + safe method (GET/HEAD/OPTIONS)
        Flask->>Validator: verify signed token
        alt valid + email matches
            Validator-->>Flask: validator_email
            Flask->>View: view(email=validator_email)
        else invalid OR mutation method
            Flask->>OAuth: fall through to OAuth
        end
    else
        Flask->>OAuth: get_current_user_email()
        alt no session
            OAuth-->>Browser: 302 to /login/google
            Browser->>OAuth: complete OAuth
            OAuth-->>Browser: redirect to /protected
        else session exists
            OAuth-->>Flask: email from token
            alt email != AUTHORIZED_EMAIL
                Flask-->>Browser: 403 unauthorized
            else
                Flask->>View: view(email=email)
            end
        end
    end
    View-->>Browser: response

Participant labels short for layout (#43). Real callers: Flask = the @login_required wrapper in auth.py. Bypass = _dev_bypass_active(), the four-gate local- only short-circuit (ADR-002). Validator = validator_cookie.py, signed long-lived token narrow-scoped to safe HTTP methods (ADR-004). OAuth = Flask-Dance Google OAuth blueprint. View = the actual Flask view function.

Related ADRs: ADR-001 (single-user OAuth lockdown), ADR-002 (four-gate dev bypass), ADR-003 (validator cookie narrow scope, SUPERSEDED), ADR-004 (broaden validator cookie to GET-only routes).
