{# ============================================================= CAPTURING ============================================================= #}

Capture bar hints

The capture bar at the top of every main page accepts plain text for the task title, plus inline hashtag hints that set tier, type, repeat frequency, and URL in one shot. Hints are stripped from the title before the task is saved.

Default tier is Inbox. Default type is whatever the dropdown next to the input is set to (Work or Personal). Hashtag hints override both.

Tier hints

  • #today → Today
  • #tomorrow → Tomorrow
  • #week → This Week
  • #next_week or #nextweek → Next Week
  • #backlog → Backlog
  • #freezer → Freezer

Only one tier hint is honored per capture — the first match wins.

Type hints

  • #work → type Work
  • #personal → type Personal

Repeat hints

Make the task recurring. A repeat template is created automatically and linked to the task.

  • #daily — fires every day
  • #weekdays — fires Mon–Fri only
  • #weekly — fires weekly on the same day-of-week you captured it
  • #monthly — fires monthly on the same day-of-month you captured it

URL auto-detection

Any http:// or https:// substring is detected, placed in the task's URL field, and stripped from the title. Whatever's left (before + after the URL) becomes the title. If nothing's left, the URL itself becomes the title.

Matching rules

  • Hints are case-insensitive — #TODAY works.
  • Hints can appear anywhere — beginning, middle, or end of the text.
  • Longest-first matching: #next_week beats #week, #weekly beats #week.
  • Repeat hints are consumed before tier hints. #weekly on its own leaves the tier at Inbox (the repeat ate the #week substring). Add an explicit tier: #today #weekly.

What hints do NOT do

  • No project or goal hints — assign those via the detail panel or bulk toolbar.
  • No due-date hints (due:2026-05-01, @friday, etc.) — set in the detail panel.
  • No priority / notes / checklist hints.
  • Anything that isn't #-prefixed stays in the title verbatim.

Worked examples

Fix login bug #today #work
  → title: "Fix login bug", tier: Today, type: Work

Team retro #weekly
  → title: "Team retro", tier: Inbox, type: (dropdown), repeats weekly

Plan Q3 #next_week #work
  → title: "Plan Q3", tier: Next Week, type: Work

Call mom #tomorrow
  → title: "Call mom", tier: Tomorrow

Take meds #daily #personal
  → title: "Take meds", type: Personal, repeats daily

Read article https://example.com/foo #personal
  → title: "Read article", type: Personal,
    URL: https://example.com/foo

https://example.com/foo
  → title: "https://example.com/foo" (URL-only, title falls back to URL)

Voice memo vs. voice dictation

Two different voice buttons sit next to the capture bar. They look similar but behave differently — pick based on how long your input is and whether you want parsing.

🎤 Voice dictation (capture bar)

  • Uses the browser's Web Speech API — no upload, no Whisper, no Claude.
  • Appends the transcript directly to the capture-bar input so you can submit it as one task.
  • Single utterance per tap — stops listening when you pause.
  • No NLP: whatever you say becomes the task title (hashtag hints still work on the text).
  • Works best for short captures ("Fix login bug hashtag today hashtag work").
  • On iOS Safari, interim results are disabled — you see the final transcript only.

🎙️ Voice memo (dedicated page)

  • Opens /voice-memo, a record-and-review flow.
  • Records with MediaRecorder, uploads the audio to the server.
  • Server pipeline: Whisper transcribes → Claude parses the transcript into one or more task candidates.
  • Each candidate comes back with inferred title, type, tier, and due date, plus optional project/goal hints.
  • You review each candidate in a list, tweak any field, and then Add All or Add Selected.
  • Non-task items (reflections, asides) are grouped in a collapsed "Reflections" section, unchecked by default.

Size and duration limits

  • Voice memo recording cap: 10 minutes (client-side hard stop).
  • Whisper upload cap: 25 MB.
  • Global request cap for any upload: 30 MB.

Privacy

  • Voice memo audio is processed in memory only — never written to disk or database.
  • Dictation audio never leaves the browser (Web Speech API is client-side).
  • Transcripts are not persisted beyond the review screen.

When each fails

  • 🎤 dictation is unsupported on some desktop browsers and unreliable in iOS Safari PWA standalone mode.
  • 🎙️ voice memo requires OPENAI_API_KEY and ANTHROPIC_API_KEY configured on the server.
  • iOS Safari has occasionally shipped MediaRecorder bugs that break the memo flow. Fall back to typing if you hit one.

Image scan

The /scan page accepts a photo of handwritten notes, a meeting agenda, or a screenshot and turns it into task or goal candidates. Pick the mode with the radio toggle at the top.

Modes

  • Tasks — candidates land in Inbox after confirmation. One candidate per extracted line.
  • Goals — candidates become Goal rows with default category=work, priority=should, status=not_started. Edit any on the review screen before confirming.

Pipeline

  1. Google Vision OCRs the image.
  2. Claude parses the OCR text into candidates.
  3. Review screen shows each candidate as editable.
  4. On confirm, selected candidates become Tasks (or Goals), tagged with a shared batch_id so the whole scan is undoable as one unit.

Accepted formats and limits

  • Accepted MIME types: image/jpeg, image/png, image/webp.
  • Max file size: 10 MB.
  • Images are processed in memory only — never saved to disk or database.

When it's wrong

  • OCR quality depends on handwriting legibility and lighting. Typed text is nearly always accurate; cursive script misses more often.
  • Edit any candidate on the review screen before confirming.
  • Deselect a candidate's checkbox to drop it from the batch without deleting.

Pasting tasks from OneNote

The /import page accepts raw text pasted from OneNote (or any source that gives you one line per task). This section describes exactly how that text gets parsed so you can clean your source before pasting.

The one-line rule

One non-empty line = one task. The parser walks the text line by line. Every line that survives the skip rules (below) becomes one task, with the line content as the task title. There's no multi-line title, no notes field populated from adjacent lines, no nested subtasks from indentation.

What the parser strips from each line

Before using a line as a task title, these prefixes are removed:

  • Dashes: -
  • Asterisks: *
  • Unicode bullets: •, ‣, ▪
  • Numbered lists: 1., 2), 14., etc.
  • Checkboxes: ☐, ☑, ✓, [x], [ ], []
  • Leading / trailing whitespace (including tabs). Indentation is NOT significant — a four-space-indented line is treated identically to an unindented line.

What gets silently skipped

  • Empty lines.
  • Lines shorter than 2 characters after cleanup.
  • Lines that match "section header" patterns: ALLCAPS (≤40 chars), a bare date (e.g. 2026-04-20), or a "Month Day, Year" line. These usually exist as headings in your source and would be noise.
  • Exact duplicate titles within the same paste — only the first occurrence is imported (case-insensitive match).

What the parser does NOT do (yet)

Understanding what's missing is as important as what's present:

  • No inline metadata shortcuts. #today, #work, due:2026-01-15, @project and similar tags are kept in the title, not parsed. All imported tasks land in Inbox with type=work.
  • No project or goal linkage during import — assign those after review via the task detail panel.
  • No URL extraction. A line containing https://example.com/page keeps that URL in the title; the dedicated URL field stays empty.
  • No subtasks from indentation. Flat list, always.

Practical "clean up before pasting" checklist

  1. Remove date-range headings and section dividers — or trust the ALLCAPS / date-pattern skip rules to drop them. Works for most.
  2. Strip anything like "TO DO" / "IN PROGRESS" column labels if OneNote keeps them as their own lines.
  3. De-duplicate deliberately — the parser drops dupes within the same paste, but it can't dedupe against existing tasks in your DB automatically. Duplicates across imports are flagged for you to toggle off on the review screen.
  4. If you need metadata (tier, type, due date, project, goal): easiest to import first, then use the bulk-select toolbar on the resulting Inbox to apply those in one pass.

Size & undo

  • Text paste has no explicit size cap in the app, but the server request body is capped at 30 MB (you'll hit browser limits well before that matters).
  • File uploads (.docx, .xlsx) are capped at 5 MB.
  • Every import is tagged with a batch_id. If you change your mind, use the Recycle Bin to undo the whole batch in one action.

Worked example

Pasted text:

APRIL 20

- [ ] Email vendors about Q3
- [x] Finalize sprint retro notes
  - Follow-up: schedule 1:1 with Dana
* Prep slides for Tuesday demo
1. Review draft PR #42
2) Read https://docs.example.com/auth
Review draft PR #42

2026-05-01

Skip this line (too short)

What gets imported (7 → 5 tasks):

  • ✅ Email vendors about Q3
  • ✅ Finalize sprint retro notes
  • ✅ Follow-up: schedule 1:1 with Dana (indentation is ignored — treated as its own task, not a subtask)
  • ✅ Prep slides for Tuesday demo
  • ✅ Review draft PR #42
  • ✅ Read https://docs.example.com/auth (URL stays in the title)
  • ❌ APRIL 20 — skipped (ALLCAPS header)
  • ❌ 2026-05-01 — skipped (date-only)
  • ❌ Second Review draft PR #42 — skipped (dupe within paste)
  • ❌ Skip this line (too short) — not skipped by length; imported normally. The name is misleading for this example.

All 6 successful tasks land in Inbox with type=work. You then bulk-select them, assign tier/project/due_date in one pass, and move on.

{# ============================================================= Section: Importing goals from Excel ============================================================= #}

Importing goals from Excel

The Excel goals importer accepts .xlsx files (5 MB max) from the /import page.

Required column

  • title (case-insensitive header match). Rows with empty title are skipped.

Optional columns

Each accepts a predefined value; anything else falls back to the default. Matches are case-insensitive.

  • category: health, personal_growth, relationships, work. Default: work.
  • priority: must, should, could, need_more_info. Default: should.
  • status: not_started, in_progress, done, on_hold. Default: not_started.
  • actions: free-text.
  • target_quarter: free-text (e.g. Q2, Q1,Q2).
  • notes: free-text.

De-duplication

Within the same file, exact title matches (case-insensitive, whitespace-trimmed) are consolidated to the first row. Duplicates against existing goals are flagged on the review screen — you choose whether to skip them.

Undo

Same batch_id mechanic as the text import. Undo the whole batch from the Recycle Bin.

Tiers and auto-fill

Every task lives in exactly one tier. Tiers determine where the task renders on the board and drive a few automatic behaviors around due dates.

Tier list (board display order)

  1. Inbox — default for new captures; unsorted queue.
  2. Today — do today.
  3. Tomorrow — do tomorrow. Auto-rolls into Today at 00:01 local time.
  4. This Week — do this week. Tasks are grouped by day-of-week inside the panel.
  5. Next Week — do next week. Same day-of-week grouping.
  6. Backlog — later but still on the list.
  7. Freezer — maybe never. Review every so often to Keep, Delete, or pull back.

Due-date auto-fill

When a task lands in Today or Tomorrow with no explicit due date, the system fills it automatically:

  • TODAY → due_date = today (local timezone).
  • TOMORROW → due_date = today + 1.
  • Other tiers → no auto-fill.

Auto-fill is fill-if-null only — it never overwrites an explicit value you set. And it skips entirely if you mentioned due_date in the same update (even setting it to blank is treated as user intent).

Moving a task back out of Today/Tomorrow does NOT clear the auto-filled date. The date is still meaningful as a reminder.

Tomorrow → Today roll

A scheduled job at 00:01 local (DIGEST_TZ) moves every Tomorrow task into Today. The idea: yesterday's "tomorrow" is today's "today," and you shouldn't have to move them by hand.

Task status (separate from tier)

  • Active — default.
  • Archived — completed. Shows on the /completed page.
  • Cancelled — consciously dropped, not done. Optional reason field.
  • Deleted — soft-deleted. Hidden from the board; not in the recycle bin (regular deletes bypass the bin).

Recurring tasks

A recurring task is a template. The template itself never appears on the board — it spawns real task instances on the days it fires.

Frequencies

  • Daily — fires every day.
  • Weekdays — fires Mon–Fri only.
  • Weekly — fires on one specific day-of-week.
  • Day of week — same as weekly; used internally for templates with a specific weekday.
  • Monthly (by date) — fires on a specific day-of-month (e.g. the 15th).
  • Monthly (nth weekday) — fires on a specific weekday of the month (e.g. first Monday).

How a template gets created

  • Capture-bar hint (#daily, #weekdays, #weekly, #monthly) — creates a template matching today's weekday or day-of-month.
  • Task detail panel — add a repeat rule to an existing task.
  • Recurring editor — directly via the /recurring editor.

When tasks spawn

A scheduled job at 00:05 local time runs through every active template and, for each one firing today, creates a real task in Today with due_date set to that fire date.

Spawning is idempotent — running the cron twice on the same day or manually triggering a spawn will not create duplicates. A planned-ahead task in This Week with matching template + due date also blocks a duplicate.

Preview cards vs. real tasks

In the This Week and Next Week panels you'll see two card styles:

  • Solid border — a real task. Has a due date, is checkable, can be edited or moved.
  • Dashed border — a preview of a recurring template that will fire on that day. Not a real task row yet. Clicking a preview opens the template editor, not a detail panel.

Previews disappear when the template fires (the real task takes the preview's place) or when a real task for the same template + date already exists.

Subtask inheritance

When a recurring parent's children are snapshotted to the template, those subtasks are cloned fresh every time the parent spawns. Snapshot captures only active subtasks at template-save time — completed or cancelled children don't carry forward.

Review mode

The /review page steps through stale tasks one at a time so the board doesn't accumulate forgotten rows.

What "stale" means

  • A task is stale if it has never been reviewed, or if it was last reviewed more than 7 days ago.
  • Completed / cancelled / deleted tasks are never stale — they're out of the queue.

Actions

One button per task. The card advances automatically after each action.

  • Keep — leave the task in its current tier. Stamps last_reviewed = today, so it drops out of the queue for 7 more days.
  • Freeze — move to Freezer and stamp last_reviewed.
  • Snooze — leave in place and stamp last_reviewed = today. Effectively the same as Keep; separate button for when you're deliberately deferring the decision.
  • Delete — soft-delete (status = deleted). Does not go through the recycle bin (that's for import undos only).

UI

  • A progress bar at the top shows how many stale tasks remain.
  • An "All caught up" screen renders when the queue is empty.

Subtasks

A task can have child tasks one level deep. Subtasks are useful for breaking a larger thing into concrete steps without leaving the parent tier.

Rules

  • One level only — a subtask cannot have its own subtasks. Attempting to parent a child to another child is rejected.
  • Goal + project inheritance — when a subtask is created without explicit goal_id / project_id, it inherits both from its parent. You can override either on the subtask itself.
  • Parent-card badge — parent cards show a badge like 2/5 subtasks. It turns green when the count hits total.

Completing a parent with open subtasks

Completing a parent with still-active subtasks shows a prompt: complete all children along with the parent, or close each subtask individually first. There's no "archive parent and leave orphans" path — that would leave subtasks with no visible home on the board.

Subtasks and recurring templates

If the parent is a recurring task, its currently-active subtasks are snapshotted into the template's subtasks_snapshot. Every time the parent spawns, the template clones those subtasks fresh. See Recurring tasks.

Navigating from subtask back to parent

The subtask detail panel shows a "← Parent" link that opens the parent's panel. If the parent is archived, cancelled, or deleted, the link still appears but is labeled with the parent's status.

Bulk operations

Multi-select + a sticky bottom toolbar lets you apply one change to many tasks in a single action.

Entering multi-select mode

  • Tap the multi-select toggle in the top-right of any tier panel.
  • Each task card shows a checkbox. Tap cards to include them.
  • The sticky toolbar appears at the bottom of the screen.

Toolbar actions

  • Type — set all selected to Work or Personal.
  • Tier — move all selected to a target tier.
  • Goal — link all selected to a goal (or clear the link).
  • Project — link all selected to a project (or clear).
  • Complete — archive all selected.
  • Delete — soft-delete all selected.

Rules

  • Max 200 tasks per call. Beyond that, the request is rejected at the API layer.
  • Best-effort semantics: per-task failures are reported individually but don't roll back successful ones.
  • Cascade rules still apply — if you move a parent to a different tier, nothing forces subtasks to follow (subtasks stay put unless you include them in the selection).
  • Subtask goal/project inheritance still applies when the parent is changed.

Recycle bin and undo

The /recycle-bin page is the undo surface for bulk imports only — OneNote pastes, Excel goal imports, and image-scan confirmations. Regular task deletes do not go through the bin.

How a batch lands there

Every import tags its new rows with a shared batch_id. When you click Undo on an import (from Settings → Import History or via the Recycle Bin), every row with that batch_id is soft-deleted in one step.

Batch states

  1. Live — rows visible on the board.
  2. In bin — rows hidden. Undo moves a batch here.
  3. Purged — rows hard-deleted from the DB. The ImportLog row stays as an audit trail.

Actions on a batch in the bin

  • Restore — put the rows back on the board.
  • Purge — permanent delete. Requires typing DELETE in the confirmation dialog.

Empty Bin

Purges every batch currently in the bin in one step. Also requires typing DELETE.

No automated cleanup

Nothing in the bin is ever deleted on a timer. Items stay there until you purge them. A TTL-based auto- expiry is in the BACKLOG Freezer as a deferred idea.

Scope reminder

  • Regular delete (trash icon on a task card) → soft-deletes immediately, does NOT go to the recycle bin.
  • Review-mode Delete → same; does NOT go to the recycle bin.
  • Only bulk-import batch undos end up here.

Print view

The /print page renders a server-side print-optimized page for taking to a meeting or a coffee break without a device.

What prints

  • Overdue — tasks with a due date before today, not archived.
  • Today — everything in the Today tier.
  • This Week — everything in the This Week tier.

What doesn't

  • Navigation chrome, capture bar, action buttons — hidden via @media print CSS.
  • Inbox, Tomorrow, Next Week, Backlog, Freezer, Completed. If you need to print a specific tier, use the browser's Print to PDF after navigating there.

Layout

  • Checkbox column on the left for pen-and-paper completion.
  • Task titles with type, project, goal metadata inline.
  • Checklists (if any) render as nested checkboxes under the parent task.
  • Overdue tasks are visually highlighted.

Email digest

A plain-text daily digest is emailed via SendGrid. The schedule and contents are fixed; adjustments happen through environment variables on the deploy.

When it fires

  • Time: DIGEST_TIME env var, default 07:00.
  • Timezone: DIGEST_TZ env var, default America/New_York.
  • Recipient: DIGEST_TO_EMAIL env var. No digest is sent if this is unset.

What's in it

  1. Today's tasks (everything in the Today tier).
  2. Tasks due today from any other tier (excluding those already listed above).
  3. Overdue tasks (due date before today, not archived).
  4. Goals that have active tasks in Today.
  5. Tomorrow count and This Week remaining count.

Format

  • Plain text only — no HTML. Renders reliably in corporate Outlook and any other email client.
  • Task content is sanitized (tabs, carriage returns, excess whitespace stripped) before going into the body.
  • No API keys or session tokens ever appear in the digest body or server logs.

Preview and send-now

From /settings, the Daily Digest panel offers:

  • Preview Digest — renders the current digest inline without sending.
  • Send Now — sends the current digest immediately to DIGEST_TO_EMAIL, independent of the schedule.

Settings

The /settings page is the operator dashboard — stats, service status, digest controls, and import history.

App Overview (stats)

  • Active Tasks — count of tasks with status = active.
  • Total Tasks — every task row, including archived, cancelled, and deleted.
  • Active Goals — goals with is_active = true.
  • Recurring Templates — active recurring templates.

External Services

A status table shows whether each integration has its API key configured. The actual key values are never displayed.

  • Google Vision — powers OCR for image scan.
  • Anthropic — powers Claude parsing for image scan and voice memo.
  • SendGrid — powers email digest delivery.

Daily Digest controls

See Email digest above. The panel shows the configured recipient and From address, plus Preview and Send Now buttons.

Import History

Every bulk import is logged here with source, item count, date, and per-batch Undo/Restore actions. The Recycle Bin link shows a count of batches currently in the bin.

Quick Actions

Shortcut buttons to Weekly Review, Scan Image, Import, and Print View.
