# ADR-031: Global error handler for uncaught exceptions

Date: 2026-04-25
Status: ACCEPTED

## Context

The original BACKLOG #50 was scoped narrowly: the digest send endpoint
returned a hardcoded `"Failed to send — check SENDGRID_API_KEY"`
message regardless of what SendGrid actually reported. During the
2026-04-24 SendGrid debugging session this misled the user into
refreshing the API key twice when the real cause was a verified-
sender problem.

User explicitly asked to **scope #50 broader**: shape ALL uncaught
exceptions into proper JSON responses, not just SendGrid's. This
addresses two related symptoms:

- **#50 original**: hardcoded misleading SendGrid error
- **#52 surface**: blank `"Save failed:"` because
  `psycopg.errors.InvalidTextRepresentation` had no JSON body — the
  frontend's `apiFetch` fell back to empty `statusText`

Common root cause: every API route catches the specific exceptions
**it** expects (`ValidationError`, "not found", JSON parse errors)
and returns nicely-shaped JSON. Anything outside that expected set
escapes, hits Flask's default exception handling, and the user sees
an opaque 500 with NO JSON body.

## Decisions

### 1. Register global Flask error handlers in a new `errors.py` module

Two handlers, both wired via `register_error_handlers(app)` called
from `create_app()` AFTER blueprints are registered (so route-level
handlers still take precedence).

- `@app.errorhandler(HTTPException)` — Werkzeug's framework-raised
  4xx/5xx (404 unknown route, 405 wrong method, 413 oversize body,
  etc.). Preserve the status code; shape body as JSON for `/api/`
  paths only (HTML routes keep Flask's default error page).
- `@app.errorhandler(Exception)` — catch-all for everything else.
  Logs the full traceback (already happens via `DBLogHandler`).
  Returns sanitized JSON with the exception type, sanitized message,
  and `request_id` for log correlation.

### 2. JSON only for `/api/` paths

Wrapping every HTML 500 in JSON would break the browser's default
error page. HTML routes get Flask's stock behaviour (which can be
templated later if we want a custom error page — not needed for v1).

The check is a one-liner: `if not request.path.startswith("/api/"):
return e`.

### 3. Specific status codes by exception type

The catch-all handler walks the exception type to pick a sensible
HTTP status:

| Exception type | Status | Rationale |
|---|---|---|
| `EgressError` | 502 Bad Gateway | The request was valid; an upstream we depend on (SendGrid, Whisper, Claude, etc.) failed |
| `sqlalchemy.exc.DataError` | 422 Unprocessable Entity | The request was structurally valid but the data violated a DB constraint (e.g. invalid enum value — bug #52) |
| `sqlalchemy.exc.IntegrityError` | 422 | Same family — FK/uniqueness violations |
| Anything else | 500 Internal Server Error | Genuine server bug |

### 4. Sanitized message extraction

The user-facing message is generated by `_shape_message(e)`:

- **EgressError**: pass through verbatim. `egress.safe_call_api`
  already produces clean messages (`"SendGrid returned HTTP 403:
  ..."` per ADR-023) — preserve them.
- **SQLAlchemy DataError / IntegrityError**: walk to `e.orig` (the
  underlying psycopg/database driver exception), strip to first
  line, prefix with "Database error: ".
- **Anything else**: `f"Server error: {first_line[:200]}"`. Never
  leak full stack traces, file paths, or query strings to the user
  — those go to logs.

### 5. Refactor `digest_service.send_digest` to raise EgressError

The original `send_digest()` caught Exception → returned False, which
killed all SendGrid error context. `digest_api` then converted the
False to a hardcoded "check SENDGRID_API_KEY" string. End-user saw
an actively misleading message.

New flow:

- `_sendgrid_send` extracts the SendGrid SDK exception's
  `.status_code` + `.body` and raises `EgressError("SendGrid returned
  HTTP <code>: <body excerpt>")`
- `send_digest` no longer catches — propagates the EgressError
- `digest_api.send` no longer hardcodes a misleading message — the
  global handler shapes the EgressError into a 502 JSON response with
  the actual SendGrid detail
- The cron job (`_send_scheduled_digest` in app.py) DOES catch the
  EgressError so it doesn't crash the scheduler thread — logs +
  swallows. The error still lands in `app_logs` via DBLogHandler.

### 6. `request_id` in every error response

Every JSON error includes the `request_id` stamped by
`logging_service`. The user (or a future support flow) can paste this
into `/api/debug/logs?request_id=<id>` to find the matching server
log row, including full traceback.

## Consequences

**Easy:**
- The class of "blank error" bugs is gone. The user always sees
  something useful — at minimum "Server error: <type>" with a
  request_id.
- Adding a new external API caller via `egress.safe_call_api`
  automatically gets clean error reporting via the EgressError path —
  no per-endpoint "Failed to send" hardcoded message needed.
- Future debugging: every error is correlatable via request_id.

**Accepted trade-offs:**
- The catch-all handler hides the full traceback from users (it's in
  logs only). A DEV-mode flag could surface the traceback inline for
  local development; deferred for now since the request_id correlation
  is sufficient.
- The 422-vs-500 split for DataError/IntegrityError assumes the user
  could have prevented the error (e.g. by sending valid data). That's
  not always strictly true — could be a server-side enum-drift bug
  like #52 where the user did nothing wrong. Acceptable trade-off:
  the message is honest about the problem regardless of the status
  code, and the user typically can't tell what HTTP status came back
  anyway.
- HTML routes still use Flask's default error pages. Templating those
  is a separate ship if/when needed.

## Alternatives considered

- **Custom exception types per route family** (e.g. `TaskNotFound`,
  `ProjectInvalid`): rejected. The per-route handlers already exist
  for known error classes; the gap is the unknown classes. A new type
  hierarchy adds boilerplate without preventing the bug.
- **Make `apiFetch` on the client display the full response body when
  there's no `error` field**: rejected. Treats the symptom (frontend
  fallback) instead of the cause (server returns no JSON). The user
  would then see raw HTML in their alert which is worse, not better.
- **Sentry integration**: deferred. Sentry would centralize the
  errors but doesn't help the user-facing message. Worth doing if we
  ever go multi-user; not for a single-user app.
- **Skip the digest_service refactor; only ship the global handler**:
  rejected. The global handler can't help if the service swallows the
  exception before it's raised. Refactoring digest_service is the
  whole point — to STOP swallowing.

## Verification

- `tests/test_global_error_handler.py` (8 tests):
  - `TestHTTPExceptionHandler`: 404 on `/api/` returns JSON; 404 on
    HTML path keeps Flask's default
  - `TestUncaughtExceptionHandler`: RuntimeError → JSON 500 with
    type + message + request_id; EgressError → JSON 502 with vendor
    message intact; DataError → JSON 422 with the underlying psycopg
    message (the literal #52 reproducer); request_id present in
    every error response
  - `TestDigestSendgridErrorPropagation`: SendGrid SDK exception
    wraps to EgressError with status code + body; `send_digest`
    propagates the EgressError instead of swallowing
- All 11 quality gates green.
- Test coverage: the #52 failure mode (blank "Save failed:") is now
  asserted-against in
  `test_sqlalchemy_data_error_returns_422` — the user-facing JSON
  contains "invalid input value", "projecttype", "PERSONAL". Cannot
  silently regress.
- `test_egress_error_returns_502_with_message` asserts the SendGrid
  case from #47/#50: status 502, message contains "SendGrid", "403",
  "not verified" — exactly what the user couldn't see before.
