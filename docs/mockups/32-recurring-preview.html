Mockup — Backlog #32

Preview recurring templates in This Week, grouped by fire day.

Scenario baseline: today is Monday. You have two real tasks in This Week (a Tuesday dentist + a Thursday report). You also have three recurring templates that will fire later this week: Weekly review (Friday), Team retro (Friday), and a daily Exercise task (every day including today, but the today instance has already spawned so only Tue–Sun are previewed).

What to look for: Scroll through the three treatments below. Each shows the same This Week panel with the same preview cards — only the visual treatment differs. The real tasks (dentist / report draft) should stay visually dominant; the preview cards should read as "coming up, not actionable yet" at a glance.

A Dashed border

Minimal visual weight. "Unfilled outline" reads intuitively as "placeholder / not yet a real thing." Uses color only sparingly (grey). Scales to 5+ previews per week without getting noisy.

C Muted opacity (~55%) + right-edge 🔁 icon

"Faded" idiom. Cards look visibly less alive than real tasks — closest to a classic "placeholder / preview" pattern. The 🔁 icon gives a quick visual scanner. Risk: accessibility / color contrast — faded grey text on white may fail WCAG contrast for low-vision users.

Open design questions (for the ADR)

  1. Click behavior: clicking a preview card should open the RecurringTask editor (not a task detail panel), because there's no Task row yet. Confirm? The tooltip shown on hover (treatments A and C use the title attr) explains "recurring — click to edit the template."
  2. Drag / done / tier moves: preview cards must reject drag, check-off, and tier-button clicks (no Task ID to target). Visually, treatments A and C already read as "not draggable"; treatment B might fool the user into trying. Need explicit no-op with a subtle toast / inline hint.
  3. Same-day collision: if today is Friday and the Weekly review template has ALREADY spawned (a real Task with recurring_task_id set exists for today), we must NOT also render a preview for Friday — one or the other. The logic: if any spawned Task this week has recurring_task_id == template.id, drop the preview for that fire day.
  4. Completed templates: if the user disabled a RecurringTask (is_active = false), no preview. Straightforward.
  5. Cross-tier visibility: should Next Week show previews too? My instinct YES — the same code renders in both panels, keyed on the fire day relative to the tier's range. Gated behind the same render path so implementation cost is near zero.
  6. Count badge: should .tier-count include previews or just real Tasks? If it includes previews, the user sees "This Week 7" and expects 7 clickable-doable things, which is wrong. Recommend: exclude previews from the count (the Friday "3" in the mockups is real tasks only in the current drafts — I need to fix that in the preview). Or show as 4 (+3 upcoming).
  7. Mobile (375px): all three treatments fit the card width (no additional width cost vs real cards). Worth verifying with a real render on mobile — not done in this static mockup.
Next step: pick a treatment and answer questions 1–6. I'll then write ADR-015 (or fold into a rev of ADR on #32) and scope the implementation. Anticipated effort: medium — the compute side is straightforward (expand templates 7 days ahead, filter already-spawned, merge into renderBoard); the UX is where the decisions live.
