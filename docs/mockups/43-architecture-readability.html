The same six pieces, written out:

  1. The web app — a Python service (Flask) that serves you the HTML pages and handles your clicks. Runs inside Gunicorn workers on Railway.
  2. The database — Postgres (managed by Railway). Stores your tasks, goals, projects, etc. See the schema below for what's in it.
  3. Scheduled jobs — four automated tasks that run on a clock (in your timezone, default America/New_York):
    • 06:30 daily — sends you the digest email
    • 00:01 nightly — moves Tomorrow tasks to Today
    • 00:05 nightly — spawns the day's recurring tasks
    • every 45 seconds — heartbeat (proves the scheduler is alive to other workers)
  4. Authentication — Google sign-in. Only one email is allowed in (the one configured on Railway); everyone else gets a 403 page. There's also a long-lived signed "validator cookie" used by automated post-deploy checks.
  5. External services — OpenAI Whisper (transcribes voice memos), Anthropic Claude (extracts tasks from voice memos and image scans), Google Vision (OCRs images), SendGrid (delivers the digest email).
  6. Hosting — Railway. HTTPS enforced. Health checks every few seconds via /healthz.

Step by step:

  1. At 00:05 every night, an automated job wakes up.
  2. It calculates today's date in your timezone (default America/New_York — using the server's UTC clock would put 10pm-ET spawns on tomorrow's date).
  3. It looks up which recurring task templates are scheduled to fire today (e.g. a weekly "Walk dog" template is included on Tuesdays only).
  4. For each one, it checks whether you already have a task for today's date — across all tiers, not just Today. So if you'd already moved "Walk dog" into "this week" with a due date of today, the job sees it and skips spawning a duplicate.
  5. For the rest, it creates a new task in the Today tier with today's due date.
  6. If the recurring template has subtasks, it creates those too (also in Today).
  7. You wake up, open the app, and the day's recurring tasks are waiting in Today.

Step by step:

  1. You hit the 🎙️ voice memo icon on the capture bar and start talking.
  2. Your browser records the audio (10-minute hard cap).
  3. When you stop, the audio file uploads to the server.
  4. The server checks the file (correct format, not too big, not empty), then forwards it to OpenAI's Whisper for transcription.
  5. Whisper sends back the transcript as plain text (e.g. "Tomorrow I need to pick up prescriptions and call mom about Sunday").
  6. The server then sends the transcript to Anthropic's Claude, along with a list of your active project and goal titles, asking it to identify discrete tasks and infer metadata: tier, due date, project hint, goal hint, and whether each item is actually a task or just a reflection ("I felt tired today" isn't a to-do).
  7. Claude returns a list of candidates. The server resolves any project/goal hints to real IDs (case-insensitive title match against your active projects/goals).
  8. You see a review screen: each candidate is an editable row (title, tier, date, project, goal). Reflections (non-tasks) are tucked into a collapsed section at the bottom, unchecked by default.
  9. You uncheck what you don't want, edit anything wrong, then hit Confirm.
  10. The server creates real tasks in your database with the fields you confirmed. They show up wherever you set the tier (Today, This Week, etc.).

Same three paths, but with the four-gate dev bypass check broken out (ADR-002), the validator cookie's signature/expiry verification shown explicitly (ADR-003 + ADR-004), and the OAuth callback + token storage step made visible.

The same logic, written out:

1. Is this local development with the bypass turned on?

YES → skip authentication entirely. Log you in as the configured email. Used only on your local dev machine when the .env.dev-bypass file exists. Has four safety gates including a refusal to run if any Railway environment variable is set, so it can't accidentally activate in production.

NO → continue to check #2.

2. Do you have a "validator cookie" AND is this a safe HTTP method?

(A "validator cookie" is a long-lived signed token used by the post-deploy automation to read pages without going through OAuth. "Safe" methods = GET, HEAD, OPTIONS — i.e. reading data, not changing it.)

YES, AND the email in the cookie matches your authorized email → log you in via the cookie. Used by the post-deploy Playwright tests and curl spot-checks.

NO (cookie missing/invalid OR you're trying to do something that changes data) → continue to check #3.

3. Do you have a Google OAuth session?

NO session → redirect you to Google to sign in, then come back to the page you wanted.

YES session, AND the Google account email matches your authorized email → log you in. (Most normal use is this path.)

YES session, but the email doesn't match → 403 Unauthorized page (the single-user lockdown). This is what happens if anyone else tries to sign in.

In all three paths, once authenticated, the page renders normally with your email available to the view.
