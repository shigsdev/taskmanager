| 56 | **Eliminate post-deploy hard-refresh requirement for users (SW staleness)** — user-flagged 2026-04-25 after #55: "a normal web app shouldn't require this." Diagnosis: existing SW (`static/sw.js` + `templates/base.html` handshake) DOES auto-update via 60s poll + visibility-change + initial load, BUT there's a race window. If a user navigates to a page right after a deploy AND before the handshake has detected the new SW version, the old SW intercepts the request: it serves NEW HTML via network-first, but the page's referenced static assets (CSS/JS) come cache-first from the OLD CACHE_NAME — leaving the user with new-HTML-against-old-static-asset combo. Two fix paths: (A) **drop HTML caching** in the SW entirely — `cache.put` for text/html responses removed; HTML always comes fresh from network. Removes the offline benefit for HTML (questionable since pages need API data anyway) but eliminates the race. (B) **skipWaiting on install** — new SW activates immediately when downloaded; old SW dies. Existing handshake's `userIsBusy()` guard would be removed (or kept as "wait for blur" only). Risk: user mid-edit gets a SW swap-out, possibly losing in-flight requests. (A) is safer + cleaner. | 2026-04-25 | OPEN — needs scope decision (A vs B vs A+B). Recommend A. |
